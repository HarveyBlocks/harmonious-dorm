# 引入AI, 如何呢?

模型采用GLM 4.7 Flash, 开源模型

- [ ] 存储一个宿舍上下文的聊天记录, 作为Mememory(近几条/几kb/mb的滑动窗口), 用追加写写入文件, 然后发给LLM. 如果一天的聊天记录少, 则好几天存在一个memory里, 如果memory, 且用户迟迟不删除, 则优先删除最旧的memory(按照天为单位)
- [ ] 搞一个长期记忆的memory, 用来存储一些长期记忆, 用户可以选择让哪些记忆存在长期记忆里. 长期记忆的文件更小, 更简练. 一般情况下, 就是追加写. 一段时间后, 长期记忆过大可以让LLM进行总结和文本压缩然后替换掉长期记忆(惰性判断, 异步完成). 长期记忆用于存储例如宿舍成员信息(这个在创建群聊时即可使用)
- [ ] 长期记忆, 可以通过指令来设置, 比如
  ```text
  @Robot /long-memory 设定上Robot的身份是一只猫娘JK
  ```
  或者在设置界面设置
- [ ] 短期记忆被淘汰后, 会被进行和长期记忆合并, 然后交给AI总结
- [ ] 需要流式输出吗? 不需要了吧? 因为流式输出基于Http, 不基于websocket.
- [ ] 在system 中提醒AI, 输出不超过800个字符, 注意精炼
- [ ] 系统提示词
  ```markdown
  你是一个群聊助手，需要根据整个对话历史理解用户的最新问题。注意：
  - 对话中可能有多人发言，你需要综合所有信息。
  - 如果最新问题中缺少明确的时间或地点，请从历史中推理。
  - 回答前先思考：用户想查什么？地点？时间？从历史中找线索。
  ```
- [ ] 短期记忆和长期记忆
  - [ ] 短期记忆由messages构成的聊天记录, 由客户端提供(或者进一步封装, 客户端请求聊天个数, 服务端进行消息的获取和聚合), 需要携带, 说话用户名, 用户ID, 说话内容
  - [ ] 长期记忆有两部分, 一部分是元数据, 一部分是对聊天记录的整理概括, 由服务端提供, 用户可以设置修改
  ```json
  {
    "robot":{
      "name": "萝卜头", // 舍长可设置
      "customer": [// 舍长可设置
        {
          "field": "人设",
          "value": "可爱的猫娘JK"
        },{
          "field": "语气",
          "value": "娇嗔的"
        }
      ]
    },
    "members":[
      {
        "id": 23,// 服务端提供
        "name": "宿舍成员A",// 服务端提供
        "leader": true,// 服务端提供
        "description": "最帅" // 可用户修改, 舍长可以改全员, 成员可以改自己
      },
      {
        "id": 24,
        "name": "宿舍成员B",
        "leader": false,
        "description": "最可爱"
      }
    ],
    "long-memory": "长文本, 交给AI总结, Maybe markdown, 可供用户编辑" // 舍长可以修改
  }
  ```
  
- [ ] 限额, 每个宿舍每小时可以使用AI 20 次, 每天可以使用100次
- [ ] 限制LLM的token数量
- [ ] 允许用户使用@ 来召唤机器人
- [ ] 允许用户进行聊天
- [ ] 允许用户决定, 取几日之后的Memory作为memory(可以指定日期)
- [ ] 宿舍长可以给AI设置名字, 到时候@出来, 就需要@AI的名字了, 默认Robot. 宿舍长还可以给AI设置头像
- [ ] 给用户展示memory的用量是百分之几, 如果到了上限, 提醒用户清空memory或保留memery到几月几号.
- [ ] 增加markdown语法的渲染, 调用库
- [ ] 用户来决定短期记忆的会话条数是多少 5-30可选
  - [ ] 5-10 简单聊天
  - [ ] 10-15 高强度聊天使用(更长的记忆里, 更好的效果)
  - [ ] 15-20 技术交流
  - [ ] 20-30 超活跃群聊
- [ ] 限制用户输出长度200中文字符/400英文字符(总之不超过512Byte)
