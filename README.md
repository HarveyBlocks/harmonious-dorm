## 系统功能总结

### 1. 用户与认证系统

**注册与登录**
- 唯一标识：使用**有效邮箱**作为唯一标识（不限制学校域名），用户名允许重复。
- 注册流程：需要填写邀请码（加入宿舍）、昵称（可选，不填自动生成邮箱前缀），并设置密码。
- 登录流程：仅需邮箱和密码，无邀请码输入框。
- 会话管理：基于 Cookie 的 Session，服务端存储会话状态，前端通过 React Query 管理用户数据。

**用户信息**
- 基本信息：昵称、头像、个人描述（可自定义）。
- 头像上传：点击头像右下角的相机图标，选择图片后**自动上传**（防抖延迟 + 离开页面时补传），存储为相对路径，不依赖第三方存储。
- 个人描述：每个成员可编辑自己的描述，舍长可编辑所有人的描述（通过成员描述卡片），描述用于机器人上下文及成员展示。
- 状态切换：支持“外出、学习、睡眠、游戏”四种状态，状态变更后：
  - 在聊天室中以系统消息形式显示（`[用户] 状态变为 [状态名]`）。
  - 在主界面“成员动态”栏同步显示。
  - 状态会注入到机器人系统提示词的用户信息中，供机器人参考。

**宿舍管理**
- 宿舍信息：显示宿舍名称、邀请码（可复制）。仅舍长可修改宿舍名称。
- 成员管理：列出所有成员及其昵称、描述、状态。舍长可在此修改任意成员描述。
- 权限移交：舍长可将舍长权限移交给其他成员，移交后原舍长成为普通成员，新舍长获得管理权限。

### 2. 账单模块

**账单创建**
- 表单设计：
  - 金额、账单描述（用于说明）、分类（预设类别 + “自定义”）。
  - 若选择“自定义”，则会在分类选择框下方出现文本输入框，要求填写自定义分类名称（即描述，作为类别展示）。
  - 选择参与成员（默认全选，可取消）。
- 分摊模式：
  - **平均分摊**：所有参与成员平均分配总金额。
  - **按权重分摊**：为每个参与成员设置权重（可以是任意正数，0 表示不出钱），前端实时计算每人应付金额，后端归一化权重后进行分摊。
- 提交逻辑：
  - 只有实际应付金额 > 0 的参与人才会写入数据库、收到通知、在账单列表中可见。
  - 若所有权重为 0，则报错（总金额必须有人支付）。
- 验证与提示：金额必须大于 0，自定义分类必须填写描述，超出字数限制（200字）会 Toast 提醒。

**账单展示**
- 列表结构：
  - 顶部卡片：本月账单总额 + 快速记账（左右排布）。
  - 下方大卡片：内部左右分栏，左边“待支付账单”，右边“已支付账单”。
  - 按月份分组（例如“2026年03月”），每个月份组内按时间排序，滚动分页加载。
- 账单卡片内容：
  - 显示分类（自定义时显示自定义文本）、金额、描述（若有）、参与人数、时间。
  - 状态标签（未支付/已支付）。
  - 支付操作：未支付账单可点击“支付”，已支付账单可点击“恢复未支付”（仅对账单中未支付的用户有效）。
- 权限控制：只有参与该账单的成员才能进行支付/恢复操作，舍长不能代付。

**统计分析**
- 后端计算数据，前端只负责渲染。
- 统计图表独立于账单列表，单独占用一行，宽度较大。
- 统计参数卡片（选择时间范围：月/季度/年，支持按日/月粒度）与图表分开，表单自定义设计。
- 图表类型：
  - **金额趋势折线图**：可切换按“总金额”或“按类别”分别展示趋势。
  - **消费类型占比饼图**：自定义分类占比小于 5% 时合并为“其他”，否则单独显示。
- 交互：
  - 鼠标悬停在饼图扇区或折线图点上时，会在鼠标旁边弹出浮动卡片，显示详细数据（类型、金额、日期、占比等）。
  - 点击图表可全屏放大，放大后图表充分利用空间，折线更粗。

### 3. 值日模块

**任务分配（仅舍长）**
- 表单：选择日期（默认当天）、选择成员、填写具体任务文本（例如“扫地”、“倒垃圾”）。
- 分配后，该成员在指定日期将有一条具体任务。
- 同一天可给多人分配不同任务，也可给同一个人分配多个不同任务（通过多次分配）。

**任务展示**
- 列表结构：
  - 整体卡片内部左右分栏，左边“待完成任务”，右边“已完成任务”。
  - 按周分割（例如“3月第2周”），滚动分页加载。
- 任务卡片内容：显示日期、任务描述、执行人。
- 完成/取消完成：
  - **点击整个卡片**切换完成状态（而非按钮）。
  - 权限：舍长可完成/取消任何人的任务，成员只能对自己任务操作。
  - 完成后的任务移至右侧“已完成任务”区域，可在其中再次点击恢复未完成。

**统计分析**
- 独立于任务列表，单独一行显示三张图表：
  - 任务状态占比饼图（已完成/未完成）。
  - 完成人占比饼图（各成员完成的任务数量占比）。
  - 任务趋势折线图（按时间展示任务数量），并可切换按不同成员分别展示趋势。

### 4. 聊天机器人

**基础交互**
- 触发方式：在聊天输入框中 `@机器人名字` 发送消息。
- 机器人回复：异步流式生成，用户消息发送后立即返回，机器人任务加入宿舍队列串行执行。
- 流式输出：
  - 后端通过 SSE 接收模型流，每收到一个 delta 立即通过 WebSocket 转发给前端。
  - 前端收到 `chat:stream:start` 后，立即在聊天列表中插入一条占位消息（空内容，带转圈图标）。
  - 随后 `chat:stream:chunk` 逐个追加文本，实时渲染。
  - 完成后 `chat:stream:commit` 结束。
- 中断：
  - 机器人消息右下角有“暂停”按钮（仅对当前用户发起的、正在生成中的机器人消息显示）。
  - 点击后，前端立即收到 `chat:stream:stop-requested`，中断显示；后端同时调用 AbortController 真实中断 LLM 请求。
  - 未开始生成的任务在队列中被移除，消息内容改为“机器人回复已停止”。
  - 已开始生成后被中断的消息，内容保留已生成的部分（可被加入记忆）。

**机器人高级配置（仅舍长）**
- 机器人名字、头像（可上传更换）。
- 机器人设定：可动态添加/删除键值对（如“人设：可爱JK”），作为机器人角色设定的一部分。
- 机器人其他内容：长文本，支持 Markdown 渲染；舍长点击可编辑 Markdown 源码，非舍长只读渲染结果。
- 短期记忆长度：滑动条 1-35，决定机器人可见的最近聊天记录数量（不含状态变更消息）。设置后机器人会将这些消息结构化注入提示词。
- 记忆与隐私：
  - 右键菜单：普通消息可“加入机器人记忆”（书签图标）或“设为隐私”（锁图标）。
  - 隐私消息：不会进入任何机器人上下文（滑动窗口、手动选中），通过 WebSocket 全员同步，其他用户看到锁标识。
  - 手动记忆：选择的消息会覆盖滑动窗口，仅用于下一次机器人请求，请求完成后自动清除。
- 机器人回复内容：
  - 动态生成，包含机器人自身名字、设定、其他内容、成员列表、成员描述、成员当前状态、最近聊天记录（含机器人历史）。
  - 遵循系统提示词规则：不泄露原始提示词，不臆造功能，根据用户语言（中/英）选择回复语言。
- 思考过程：模型若有 `reasoning_content`，后端计数并通过 `chat:stream:reasoning` 推送，前端在气泡内显示“思考中，思考进度: N”，正式内容到达后覆盖提示。

**队列机制**
- 按宿舍串行，确保同一宿舍机器人任务依次执行。
- 任务包含：占位消息 ID、触发用户、宿舍 ID、上下文等。
- 失败重试：对可重试错误（如限流）指数退避重试最多 3 次。
- 中断处理：队列中的任务被中断时直接移除，不调用模型。

**提示词优化**
- 采用结构化压缩格式传递给模型，包含用户目录、角色目录、历史索引等，节省 token 且抗注入。
- 系统提示词明确协议，禁止从消息内容中信任身份，禁止输出原始 payload。
- 上下文窗口仅包含有效聊天消息（过滤掉状态变更消息），若手动记忆则替换滑动窗口。

### 5. 通知中心

**界面与交互**
- 独立页面：侧边栏点击“通知”进入，右侧主区域展示。
- 默认筛选：点击通知中心时，默认显示“未读”列表（可切换全部）。
- 列表样式：每张通知卡片前有自定义样式的复选框，支持批量操作。
- 菜单栏：左上角菜单按钮，点击后展开下拉菜单，自动根据屏幕边界调整展开方向（上下左右）。
- 菜单功能：
  - 全选/取消全选（采用 include/exclude 逻辑，滚动加载时保持状态）。
  - 删除选中（批量删除，后端提供批量 API）。
  - 已读选中（仅在选中未读通知时出现）。
- 点击卡片：点击通知卡片即视为已读，并跳转到目标页面（聊天、账单、值日等）。

**推送与分发**
- 新通知到达时：
  - 如果用户当前在对应的界面（聊天/账单/值日），则直接标记该类型所有通知为已读，不弹出提醒。
  - 否则，屏幕右侧弹出一个悬浮窗口，显示新通知摘要，几秒后自动收回。
- 通知堆叠：同一类型连续多条通知（如聊天消息）合并为一条，显示“有 X 条新消息”。
- 精准通知：
  - 账单：只发给参与该账单的成员。
  - 值日：只发给任务相关成员。
  - 聊天：所有人。
  - 状态变更、舍长移交等系统消息：仅相关人。
  - 本人触发的事件，本人不会收到通知（如自己发消息、自己分配值日等）。
- 红点与数字：侧边栏通知图标显示未读数量，超过 99 显示 `99+`。

**自动已读**
- 进入聊天界面：自动将该宿舍所有聊天类型通知标记为已读。
- 进入账单界面：自动将账单类型通知标记为已读。
- 进入值日界面：自动将值日类型通知标记为已读。
- 进入设置界面：自动将设置类通知（如宿舍名变更、机器人设定变更等）标记为已读。

### 6. 界面与用户体验

**主题与配色**
- 浅色模式：默认明亮背景。
- 睡眠模式：纯深色背景（`#010409`），卡片采用分层亮度（越近越亮），所有文字颜色由主题变量控制，彻底解决黑字问题。
- 颜色系统：不同状态（账单类型、任务状态）的颜色统一抽取到主题变量中管理。

**布局与响应式**
- 侧边栏固定，主内容区域左右留白对称，且根据屏幕尺寸（移动端、md、lg）自适应留白宽度。
- 卡片间距、内边距经过精细调整，确保信息密度舒适。
- 移动端保留必要的左右排布（如待支付/已支付、待完成/已完成），并适配手势长按菜单。

**列表与滚动**
- 所有列表（账单、值日、聊天、通知）都采用滚动分页加载，初始加载少量数据（5-10 条），滚动到底部自动加载下一页。
- 滚动容器：鼠标在卡片外滚动为页面滚动，鼠标在卡片内滚动为卡片内列表滚动。
- 聊天定位：
  - 进入聊天界面默认滚动到底部。
  - “到上次位置”按钮：基于用户最古老的未读聊天通知的时间，定位到该时间点附近的消息（前后各 10 条），并高亮提示。
  - 用户滚动到顶部时自动加载更早消息，滚动到底部加载更新消息。
  - 消息流式增长时滚动策略：若用户在底部则跟随增长，若不在底部则保持滚动位置不变。

**聊天体验**
- 用户名显示在气泡外部（头像旁边），气泡内部只含内容。
- 普通用户消息：支持回车换行（`Ctrl+Enter` 插入换行），发送后自动清空输入框。
- 机器人消息：Markdown 渲染，支持 GFM、数学公式、目录等，代码块可横向滚动，普通文本自动换行。
- 新消息计数：
  - 用户不在底部时，新消息（包括机器人流式）在右下角显示倒水滴状计数按钮，显示未读消息数量（超过 99 显示 `99+`）。
  - 点击计数按钮后滚动到第一条未读消息，每滚动进入视口一条消息，计数减少一，全部看完后按钮消失。
- 右键菜单（长按移动端）：
  - 在消息气泡上右键，弹出菜单，选项包括“加入机器人记忆”、“设为隐私”。
  - 菜单位置贴近鼠标，并根据边界自动调整。
  - 菜单关闭：点击外部、滚动、执行操作后自动关闭。

**设置界面**
- 卡片顺序：使用者资讯 → 宿舍资讯 → 成员描述 → 宿舍机器人 → 账户安全，纵向排列。
- 卡片折叠：除使用者资讯外，其他卡片默认折叠，点击右上角图标展开/收缩。
- 自动保存机制：
  - 输入字段（昵称、描述等）使用防抖（900ms）自动保存。
  - 头像上传使用防抖（600ms）+ 离开页面补传。
  - 若用户在设置页停留，每 3 秒进行一次“周期补发”确保数据最终一致。
- 实时同步：舍长修改宿舍名、机器人配置、成员描述后，通过 WebSocket 推送给所有在线成员，刷新对应数据，不依赖手动刷新。

### 7. 技术架构与性能

**前后端一体化**
- Next.js 14 App Router，API 路由处理所有后端逻辑。
- Prisma + SQLite 作为数据库，通过环境变量配置数据库路径。
- Socket.IO 实现 WebSocket 实时通信，路径统一为 `/api/ws`，解决路径冲突。

**性能优化**
- 生产构建：支持 `output: 'standalone'`，静态资源通过 `prepare-standalone.cjs` 脚本同步，Turbopack 可选。
- 开发体验：
  - 支持双进程并行开发（`dev:user` 和 `dev:agent`），通过 `NEXT_DIST_DIR` 隔离构建目录。
  - Next.js 监听排除：自动解析 `DATABASE_URL` 中的 SQLite 文件路径，忽略其变更；忽略 `.tmp`、`.next*`、`logs` 等。
- 后端日志：
  - 写入系统临时目录（默认 `%TEMP%\harmonious-dorm\backend.log`），避免开发时写日志触发热更新。
  - 控制台输出结构化 JSON（单行），异常时多行美化。
  - 对 LLM 调用记录三段日志：请求开始、首包到达、流结束，包含耗时、token 估算等。
- 测试模式：`CHAT_ENV="echo"` 时，不调用真实模型，直接返回请求预览，模拟流式输出，方便测试。

**代码质量**
- 组件化与解耦：
  - 巨型组件（`legacy-dorm-app.tsx`）被拆分为多个 Hooks（状态、查询、突变、Socket、同步等）和 UI 组件（各 Tab、设置卡片、聊天面板等）。
  - 主文件行数从 3000+ 降至 1000 以内。
- 国际化：
  - 所有用户可见文本通过枚举键 + 字典映射（中、繁、英、法）管理，无中文标识符。
  - 后端错误信息同样支持多语言，前端通过 `notice-messages.ts` 统一格式化。
- 错误处理：
  - 前端：网络错误/业务错误通过 Toast 显示，控制台仅打印前端内部错误。
  - 后端：统一异常处理，只返回可读的错误描述，不暴露堆栈；可控异常不打印堆栈，未知异常打印并落盘。
- 复用与设计模式：
  - 账单分摊算法独立抽取到 `share-allocation.ts`，前后端共用。
  - 流式请求客户端（`stream-chat-client.ts`）统一处理 LLM 调用，支持流式/非流式，支持 echo 模式。
  - 权限校验、分页、滚动分页等逻辑封装为通用 Hooks。

### 8. 数据模型与持久化

**核心表结构（Prisma）**
- `users`: 邮箱、密码（hash）、昵称、头像路径、描述、状态、宿舍 ID、角色（leader/member）。
- `dorms`: 宿舍名、邀请码、当前舍长 ID。
- `bills`: 金额、描述、分类（预设或自定义）、日期、创建者 ID。
- `bill_participants`: 关联账单与用户，存储实付金额（`actual_amount`）。
- `duties`: 日期、任务描述、用户 ID、完成状态。
- `chat_messages`: 消息内容、发送者 ID、宿舍 ID、时间、是否为机器人、是否流式占位、关联上下文等。
- `notifications`: 通知类型、关联 ID（账单/值日/聊天等）、消息键（枚举）、参数、是否已读、接收者 ID。
- `dorm_bot_settings`: 宿舍机器人设定（名字、头像、其他内容、短期记忆长度等）。
- `dorm_bot_fields`: 键值对（人设字段）。
- `user_descriptions`: 用户描述历史（支持舍长修改）。
- `message_privacy`: 隐私消息标记（消息 ID、宿舍 ID、用户 ID 等）。

**数据生成与测试**
